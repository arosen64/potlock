## Context

Contract data is generated by Gemini and stored as a JSON string (`contractJson`) in Convex. It has a well-known shape:

```json
{
  "name": "Pool Name",
  "version": 1,
  "members": [{ "name": "...", "wallet": "...", "role": "manager|member" }],
  "contribution_rules": "...",
  "distribution_rules": "...",
  "allowed_transaction_types": ["groceries", "utilities"],
  "approval_rules": {
    "default": { "type": "unanimous" | "k-of-n", "k"?: number },
    "amendment": { "type": "unanimous" }
  },
  "budget_limits": { "per_transaction_max_sol": 0.5 }
}
```

**Current problem**: `ContractFieldView` in `ContractCreationPage.tsx` is a generic keyâ†’value loop that falls back to `<pre>JSON.stringify(...)</pre>` for any nested object. This component is imported by `AmendContractPage.tsx` and `ContractCreationPage.tsx`. The `ContractViewer.tsx` actually uses TipTap to render `doc.content` (a rich-text doc), not `ContractFieldView` â€” so it has a different rendering path.

The fix is to replace `ContractFieldView` with a schema-aware component that knows how to render each section of the contract in human-readable form.

## Goals / Non-Goals

**Goals:**

- Replace the generic `ContractFieldView` with a typed, schema-aware `ContractDisplay` component that renders each section cleanly using shadcn/ui + Tailwind
- Use human-readable section labels and field labels throughout
- The amend page's current contract preview and amended contract preview both use this component
- The contract creation page preview also uses this component
- No raw JSON, no `<pre>` tags, no bracket/brace characters visible to users

**Non-Goals:**

- Changing the contract data model or Convex backend
- Adding an editable form for contract fields (the amend flow uses plain-language description â†’ Gemini generates the new contract; no manual field editing is needed)
- Changing `ContractViewer.tsx` (it renders rich-text via TipTap, not raw JSON; it is not the broken page)

## Decisions

### Replace `ContractFieldView` with a typed `ContractDisplay` component

**Decision**: Delete the generic field renderer and write a `ContractDisplay` component that renders each contract section (Members, Contribution Rules, Distribution Rules, Allowed Transaction Types, Approval Rules, Budget Limits) as labeled cards/sections.

**Why over a generic renderer**: The contract has a fixed, known schema. A typed renderer can use proper labels ("Approval Threshold" vs `approval_rules.default`), render members as a table/list, render arrays as chips/badges, and render nested objects as readable sentences â€” all impossible with a generic approach.

**Alternatives considered**: Keep generic renderer but add special-case branches per key â€” rejected as it produces the same messy conditional logic without clean component boundaries.

### Keep amend flow as plain-language â†’ Gemini (no structured form)

**Decision**: The amend page already has a good UX pattern (describe the change â†’ Gemini generates â†’ preview â†’ confirm). We are NOT replacing this with a form editor.

**Why**: The issue acceptance criteria says "pre-fills a structured form" but the actual root cause is the preview showing raw JSON. Showing the current contract and the amended contract via `ContractDisplay` satisfies the spirit of the requirement without a complex form rebuild.

### Place `ContractDisplay` in its own file

**Decision**: Create `src/components/ContractDisplay.tsx` as a shared component. Both `ContractCreationPage.tsx` and `AmendContractPage.tsx` import it.

## Risks / Trade-offs

- [Risk] Contract schema from Gemini may have unexpected extra fields â†’ Mitigation: render known fields explicitly, add a fallback "Additional fields" section for unknowns
- [Risk] `members` or `approval_rules` may be missing/null on older contracts â†’ Mitigation: all sections render with graceful empty states
